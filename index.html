各模組細節

每個模組展開後的關鍵設計決策與實作細節。

Module A · SAC 巡航

SmartPatrolNode

  • SAC 演算法（Stable Baselines3），論文完整復現
  • 觀測空間 25 維：20 個 LiDAR bin（每 bin 取最小值）+ 目標距離 + cos(θ) + sin(θ) + 前一步速度
  • cos/sin 編碼避免角度週期性問題（0° 和 360° 不連續）
  • 動態巡邏點：/patrol/goto topic 即時重定向，/patrol/reload 整批更新
  • 卡住偵測：3 秒未移動 → 倒退 2 秒 + 交替左右轉向
  • 安全優先：收到 /security/alerts 立刻 v=0, ω=0
Module B · 資安監控

DDSSecurityMonitor

  • SROS2：每個節點持有 X.509 憑證，沒有憑證無法加入 Domain 30
  • DDS 加密：AES-GCM 加密訊息內容，HMAC-SHA256 驗簽名
  • permissions.xml 最小權限原則：patrol_node 只能訂閱 /scan /odom，只能發布 /cmd_vel
  • NodeScanner 每秒輪詢 DDS 圖，對比合法節點白名單（9 個）
  • 偵測到異常 → 發布 /security/alerts（RELIABLE + TRANSIENT_LOCAL）→ 停止機器人
  • 已知漏洞：白名單只比對短名，命名空間欺騙可繞過（待修補）
Module C · 任務管理

Mission & Status Nodes

  • mission_manager：訂閱警報，管理任務狀態機（巡邏 / 停止 / 恢復）
  • 警報後自動等待 30 秒恢復，可設定
  • system_status_node：對外廣播節點健康度和任務進度
  • sensor_hub_node：LiDAR + Odom 資料融合與分發
Module D · LLM 模糊測試

LLM Fuzzer

  • MutationAgent（GPT-4o-mini）：生成惡意節點名稱測資，策略涵蓋命名空間欺騙、前綴繞過
  • asyncio.Queue（容量 32）：1 Producer + 4 Consumer，Pipeline 解耦
  • Docker Sandbox：network=none，mem_limit=256m，隔離執行靶機
  • 靶機 target_monitor.py：完整複製 B3 的白名單邏輯，exit 2 = BYPASS FOUND
  • TriageAgent（GPT-4o）：分析繞過原因，輸出嚴重程度與修補建議
  • SQLite StateDB：Hash 去重防重複，支援斷點續傳

Security Findings · LLM Fuzzer

資安測試結果

用 LLM 模糊測試對 Module B 白名單邏輯進行系統性測試，共發現 31 筆繞過漏洞。

27
WHITELIST_NAME_MATCH — CRITICAL
白名單比對只看節點短名（rsplit('/', 1)[-1]）， 攻擊者在任意命名空間下使用合法名稱（如 /attacker/patrol_node）即可通過白名單，完全偽裝成合法節點。
4
UNDERSCORE_PREFIX — HIGH
節點名以 _ 開頭時，偵測邏輯直接 continue 跳過，不進入白名單比對。 攻擊者只需在節點名前加底線，即可完全隱身。
3 × 1
PREFIX BYPASS — HIGH
ros_gz、dqn、launch_ros 開頭的節點同樣被靜默略過， 各前綴各發現 1 筆繞過。原意是略過內部工具節點，但沒有任何驗證機制。

修補方向（尚未實作）

根本解法是改白名單比對為全路徑（namespace + name），並移除所有無條件略過前綴。 更強的做法是把偵測邏輯的信任來源換成 SROS2 憑證本身——有憑證才算合法，不靠名稱白名單。 目前漏洞已記錄在 Module B 的 AnomalyDetector 中，等 SAC 訓練完成後一起修補。
